### Permisos y acceso

| Permiso | Acceso |
|---------|--------|
| `productos.ver` | Listar, ver detalle, ver imagenes |
| `productos.gestionar` | Crear, editar, eliminar, gestionar categorias, eliminar imagenes |

Todas las rutas requieren `auth` + `activo`.

---

### Flujo: Crear producto

1. Usuario llena formulario (nombre, categoria, descripcion, precio_referencia, stock_actual, activo, imagenes)
2. Se valida:
   - `nombre`: requerido, string, max 150
   - `categoria`: requerido, string, max 60, **debe existir** en `categorias_producto.slug`
   - `descripcion`: opcional, string
   - `precio_referencia`: opcional, numerico, min 0
   - `stock_actual`: **requerido**, integer, min 0
   - `activo`: opcional, boolean
   - `imagenes.*`: archivo, imagen, JPG/PNG/WEBP, max **10 MB**
3. Se genera codigo automatico: `PROD-XXXX` (secuencial, zero-padded a 4 digitos)
4. Se crea el producto en `DB::transaction()`
5. Las imagenes se almacenan en `storage/app/public/productos/`
6. Redirecciona a index con mensaje: *"Producto registrado correctamente."*

#### Reglas del codigo `PROD-XXXX`
- Busca el ultimo codigo existente con formato `PROD-` ordenando por `id DESC`
- Incrementa el numero secuencial
- Garantiza que no haya colisiones (vuelve a intentar si existe)
- Solo se genera en `store()`, nunca en `update()`

---

### Flujo: Editar producto

1. Usuario modifica los campos (los mismos que crear)
2. Se valida con las mismas reglas
3. El codigo del producto **no se regenera** (se mantiene el original)
4. Se agregan nuevas imagenes (sin reemplazar las existentes)
5. Se actualiza en `DB::transaction()`
6. Las categorias disponibles incluyen todas las activas **mas la actual del producto** (aunque este inactiva)
7. Redirecciona con mensaje: *"Producto actualizado correctamente."*

---

### Flujo: Eliminar producto

1. Carga todas las imagenes asociadas
2. Elimina cada archivo fisico del disco `public`
3. Elimina el registro del producto (cascade delete elimina los registros de `producto_imagenes`)
4. **No hay soft delete** ‚Äî se elimina permanentemente
5. Mensaje: *"Producto eliminado correctamente."*

---

### Flujo: Ver detalle

- Carga la relacion `imagenes`
- Muestra todos los campos del producto
- Vista con galeria de imagenes navegable

---

### Busqueda y filtros (index)

- **Busqueda textual** (`q`): LIKE sobre `codigo`, `nombre`, `descripcion`
- **Filtro por categoria** (`categoria`): coincide con columna `categoria` (slug)
- **Filtro por estado** (`activo`): "1" = activo, "0" = inactivo, vacio = todos
- Los resultados se pagan a **10 por pagina**, ordenados por `id DESC` (mas recientes primero)

---

### Imagenes

- Multiples imagenes por producto
- Almacenadas en `storage/app/public/productos/`
- Cada imagen guarda: nombre_original, mime_type, tamano_bytes
- **Eliminacion individual** via AJAX:
  - Verifica que la imagen pertenezca al producto (seguridad)
  - Elimina archivo fisico del disco `public`
  - Elimina registro de `producto_imagenes`
  - Retorna JSON
- **Visualizacion**: servida con `Cache-Control: public, max-age=86400` (24h)
- **Resolucion de ruta**: tolera multiples formatos de prefijo (`public/`, `storage/`, `app/public/`) y busca por nombre de archivo en directorios comunes como fallback para datros legacy

---

### Gestion de categorias

Todas las operaciones via AJAX (JSON):

#### Crear categoria
- Valida nombre (requerido, string, max 120)
- Genera slug con `Str::slug()` usando underscore (`_`) como separador
- Si el slug ya existe, agrega `_2`, `_3`, etc. hasta encontrar uno unico
- Las nuevas categorias siempre se crean como **activas**

#### Editar categoria
- Valida nombre
- **Si el slug cambia**, actualiza todos los productos que usaban el slug anterior (transaccional)
- La colision de slug excluye la categoria actual (puede mantener el mismo slug)

#### Activar/Inactivar categoria
- Solo cambia el booleano `activo`
- **No modifica productos** al desactivar una categoria
- Las categorias inactivas se ocultan del formulario de productos (excepto si ya estan asignadas)

---

### Reglas de negocio clave

1. **Codigo auto-generado**: `PROD-XXXX` ‚Äî solo en creacion, nunca se reasigna
2. **Categorias por slug**: los productos referencian categorias por slug, no por ID. Si el slug cambia, todos los productos se actualizan automaticamente
3. **Ciclo de vida de categorias**: pueden estar activas/inactivas. Las inactivas persisten en BD pero se ocultan de formularios nuevos
4. **Stock**: entero sin signo, validado como `min:0`
5. **Precio**: decimal(10,2), nullable, validado como `numeric|min:0`
6. **Transacciones**: creacion y actualizacion envueltas en `DB::transaction()`
7. **Sin soft delete**: productos se eliminan permanentemente
8. **Sin FormRequest**: validacion inline en el controlador
9. **Imagenes**: hasta 10 MB cada una, formatos JPG/PNG/WEBP

### Flujo: Crear cliente

1. Usuario llena formulario (nombre_completo, documento, telefono, correo, direccion, observaciones)
2. Se valida:
   - `nombre_completo`: **requerido**, string, max 120
   - `telefono`: opcional, string, max 20
   - `correo`: opcional, string, email valido, max 120
   - `documento`: opcional, string, max 25, **unico** en tabla `clientes` (ignora propio ID en edicion)
   - `direccion`: opcional, string
   - `observaciones`: opcional, string
3. Se crea el cliente
4. Redirecciona a index: *"Cliente registrado correctamente."*

---

### Flujo: Editar cliente

- Misma validacion que crear
- La validacion de `documento` unico ignora el ID del cliente actual
- Redirecciona con: *"Cliente actualizado correctamente."*

---

### Flujo: Eliminar cliente

- Eliminacion fisica (hard delete, no soft delete)
- Los pedidos vinculados conservan `cliente_id = NULL` (no se eliminan)
- Redirecciona con: *"Cliente eliminado correctamente."*

---

### Busqueda y filtros (index)

- **Busqueda general** (`q`): LIKE sobre `nombre_completo`, `telefono`, `correo`, `documento`
- **Filtro por documento** (`documento`): LIKE sobre `documento`
- Paginacion a **10 por pagina**, ordenado por `id DESC`

---

### Consulta de documento por API externa

**Endpoint:** `GET /clientes/consulta-documento?numero=XXXXXXXX`

Usado principalmente desde el formulario de pedidos para auto-completar datos del cliente.

**Flujo:**
1. Recibe `numero`, elimina caracteres no numericos
2. Valida longitud: 8 digitos (DNI) o 11 digitos (RUC)
3. **Paso 1 ‚Äî Busqueda local:** Consulta `clientes` por `documento` exacto. Si existe, retorna datos locales (`fuente: 'local'`)
4. **Paso 2 ‚Äî DNI no encontrado:** Consulta API de **RENIEC** via Decolecta
5. **Paso 3 ‚Äî RUC no encontrado:** Consulta API de **SUNAT** via Decolecta
6. Si la API externa falla o no hay API key configurada, retorna error indicando llenado manual

**Configuracion (`config/services.decolecta`):**
- `api_key` ‚Äî token de autenticacion
- `base_url` ‚Äî por defecto `https://api.decolecta.com`
- `reniec_dni_path` ‚Äî por defecto `/v1/reniec/dni`
- `sunat_ruc_path` ‚Äî por defecto `/v1/sunat/ruc`

---

### Reglas de negocio clave

1. **Documento unico:** validado a nivel de controlador (no hay indice unico en BD)
2. **Cliente-Pedido:** un cliente puede tener multiples pedidos. Al eliminar cliente, los pedidos no se eliminan (SET NULL)
3. **Busqueda combinada:** un solo campo de busqueda textual busca en 4 columnas simultaneamente
4. **API externa:** integracion con RENIEC/SUNAT via Decolecta para auto-completar datos. Siempre consulta BD local primero
5. **Sin soft delete:** clientes se eliminan permanentemente
6. **Auditoria:** solo `created_at` / `updated_at` ‚Äî no hay `deleted_at`

### Calculo de pago (regla 50/50)

| `estado_pago` | `monto_adelanto` | `monto_saldo` |
|---|---|---|
| `pendiente_adelanto` | 0 | 100% del total |
| `adelanto_pagado` | 50% del total | 50% del total |
| `pagado_completo` | 50% del total | 0 |

El sistema solo soporta esta proporcion ‚Äî no hay otras opciones de division de pago.

---

### Flujo: Crear pedido

1. Se genera codigo automatico: `PED-XXXXXX` (secuencial zero-padded a 6)
2. Si se proporciona `cliente_id`, se auto-completan los campos del cliente desde el modelo Cliente
3. Si **no** hay `cliente_id` pero hay `documento_cliente`:
   - Busca cliente existente por documento
   - Si existe, lo vincula
   - Si no existe, **crea automaticamente** un nuevo Cliente con los datos + nota "Creado automaticamente desde pedidos."
4. Si `tipo_entrega = 'local'`, se limpian todos los campos de entrega (direccion, distrito, nombre_recibe, etc.)
5. Se validan los campos de entrega como `required_unless:tipo_entrega,local`
6. Se pueden adjuntar archivos de orden: PDF, DOC, DOCX (max 15 MB c/u)

**Validacion de creacion/edicion:**

| Campo | Reglas |
|-------|--------|
| `nombre_cliente` | requerido, string, max 120 |
| `tipo_producto` | requerido, string, max 120 |
| `tipo_entrega` | requerido, in: `local,delivery,agencia` |
| `cantidad` | requerido, integer, min 1 |
| `estado` | requerido, in: `registrado,en_produccion,listo_entrega,entregado,cancelado` |
| `archivos_orden.*` | archivos: max 15MB, tipos: pdf,doc,docx |
| Campos de entrega | `required_unless:tipo_entrega,local` |
| `cliente_id` | nullable, debe existir en `clientes.id` |

---

### Flujo: Personalizacion (`actualizarPersonalizacion()`)

Endpoint separado para gestionar el diseno del pedido.

**Validacion:**

| Campo | Reglas |
|-------|--------|
| `estado` | requerido, in: `registrado,en_produccion,listo_entrega,entregado,cancelado` |
| `estado_personalizacion` | requerido, in: `sin_iniciar,en_diseno,en_revision,aprobado,en_produccion,listo_entrega,entregado` |
| `estado_pago` | requerido, in: `pendiente_adelanto,adelanto_pagado,pagado_completo` |
| `archivos_diseno.*` | archivos: max 10MB, tipos: cdr,pdf,png,jpg,jpeg,svg,ai,eps,psd,webp |

**Guarda ‚Äî bloqueo por pago:**
- Si se intenta avanzar a `en_produccion`, `listo_entrega` o `entregado` pero `estado_pago === pendiente_adelanto`, se bloquea ‚Äî se debe registrar el 50% de adelanto primero.

---

### Flujo: Pago final (`confirmarPagoFinal()`)

Cuando se cobra el saldo y se cierra el pedido:

**Pre-condiciones:**
1. `estado` debe ser `listo_entrega` o `entregado`
2. `estado_pago` NO puede ser `pendiente_adelanto` (debe haber pagado el 50%)
3. `estado_pago` NO puede ser `pagado_completo` (no doble cierre)
4. `monto_saldo` debe ser > 0
5. `monto_total` debe ser > 0

**Transaccion:**
1. Verifica que no exista una Venta de tipo `pedido` para este pedido
2. Crea `Venta` con `tipo_venta = 'pedido'`, `monto_cobrado = monto_saldo`
3. Crea `VentaDetalle` con descripcion "Pago final pedido PED-XXXXXX"
4. **Emite comprobante** automaticamente via `ComprobanteVentaService`:
   - Si `documento_cliente` tiene 11 digitos ‚Üí factura (serie F001)
   - Si no ‚Üí boleta (serie B001)
5. Actualiza el pedido: `estado = 'entregado'`, `estado_pago = 'pagado_completo'`, `monto_saldo = 0`

---

### Archivos

| Contexto | Ruta de almacenamiento | Tipos permitidos | Maximo |
|----------|------------------------|------------------|--------|
| Orden de compra | `storage/app/public/ordenes_compra_pedido/` | PDF, DOC, DOCX | 15 MB |
| Diseno | `storage/app/public/disenos_pedido/` | CDR, PDF, PNG, JPG, SVG, AI, EPS, PSD, WEBP | 10 MB |

---

### Reglas de negocio clave

1. **Codigo auto-generado:** `PED-XXXXXX` ‚Äî solo en creacion
2. **Pago 50/50 obligatorio:** el sistema no soporta otras proporciones
3. **Auto-creacion de clientes:** si se ingresa un documento no registrado, se crea el cliente automaticamente
4. **Bloqueo por adelanto:** no se puede pasar a produccion sin haber pagado el 50%
5. **Cierre automatico:** al cobrar el saldo, se emite comprobante y se marca como entregado
6. **Snapshot de datos:** `nombre_cliente`, `telefono_cliente`, etc. son una copia al momento de crear ‚Äî no se actualizan si el cliente cambia
7. **Sin soft delete:** pedidos se eliminan permanentemente

### Flujo: Venta de stock (mostrador)

1. Usuario llena formulario con:
   - Tipo de comprobante (boleta/factura)
   - Documento del cliente (DNI 8 dig / RUC 11 dig)
   - Nombre del cliente
   - Productos: busqueda autocomplete por codigo/nombre, cantidad
2. Validacion:
   - `tipo_comprobante`: requerido, in: `boleta,factura`
   - `producto_id`: array requerido (min 1), cada uno debe existir en `productos`
   - `cantidad`: array requerido, cada uno debe ser integer >= 1
   - Si es factura: documento debe tener **exactamente 11 digitos** y nombre requerido
   - Si documento tiene longitud != 8 y != 11: error
3. Por cada linea de producto:
   - **Verifica stock suficiente** (`stock_actual >= cantidad`)
   - Usa `precio_referencia` del producto como precio unitario
   - Calcula `subtotal = round(precio * cantidad, 2)`
4. **Transaccion:**
   - Crea `Venta` con `codigo = VEN-XXXXXX`, `tipo_venta = 'stock'`, `monto_cobrado = monto_total`
   - Crea `VentaDetalle` por cada linea (snapshot del nombre del producto)
   - **Decrementa** `producto.stock_actual -= cantidad` por cada producto
   - Emite comprobante automaticamente

---

### Flujo: Cierre de pedido (pago final)

Documentado en la seccion de Pedidos ‚Äî `PedidoController@confirmarPagoFinal()`.

Resumen:
1. Crea `Venta` con `tipo_venta = 'pedido'`, vincula al pedido
2. Crea un unico `VentaDetalle` con nombre "Pago final pedido PED-XXXXXX"
3. Emite comprobante
4. **No decrementa stock** (el producto ya se produjo/consumio durante el pedido)

---

### Comprobante de venta (boleta/factura)

**Servicio:** `ComprobanteVentaService@emitir()`

1. Determina tipo: 11 digitos = factura (F001), otro = boleta (B001)
2. Calcula correlativo: max correlativo para esa serie + 1
3. Genera codigo: `{serie}-{correlativo padded a 6}` (ej: `B001-000001`)
4. Crea registro en `comprobantes_venta`

**Emision manual:** `POST /ventas/{venta}/emitir-comprobante` ‚Äî solo si la venta no tiene comprobante aun.

**PDF:** `GET /ventas/{venta}/comprobante` ‚Äî genera PDF con DomPDF. Template incluye:
- Encabezado: ARTE Y METALES, RUC 20608082329, direccion Chiclayo
- Tipo de comprobante, codigo, datos del cliente
- Tabla de productos (nombre, cantidad, precio unitario, subtotal)
- Totales: subtotal items, cobrado, total comprobante

---

### Busqueda y filtros (index)

- **Busqueda (`q`)**: LIKE sobre `codigo`, `cliente_nombre`, `tipo_venta`
- **Filtro por tipo (`tipo`)**: exact match en `tipo_venta`
- Paginacion a 12 por pagina, ordenado por `id DESC`

---

### Reportes

**Filtros:** fecha_inicio, fecha_fin, tipo

**KPIs del modulo de reportes:**
- `total_vendido`: suma de `monto_total`
- `total_cobrado`: suma de `monto_cobrado`
- `ticket_promedio`: promedio de `monto_total`
- `cantidad`: conteo de ventas

**Dashboard:**
- `ventas_hoy`: suma de `monto_cobrado` del dia actual
- `ultimasVentas`: ultimas 5 ventas
- `ventas_14_dias`: ventas diarias de los ultimos 14 dias (Chart.js)
- `pedidos_pendientes`, `pedidos_produccion`, `pedidos_listo_entrega`

**Exportaciones:** CSV (UTF-8 BOM, punto y coma) y Excel XLSX (OpenXML sin librerias externas)

---

### Reglas de negocio clave

1. **Codigo auto-generado:** `VEN-XXXXXX`
2. **Stock solo en venta directa:** las ventas tipo `stock` decrementan inventario; las de tipo `pedido` no
3. **Siempre pagado completo:** las ventas no soportan credito o pago parcial ‚Äî siempre `pagado_completo`
4. **Comprobante obligatorio:** toda venta emite comprobante (boleta o factura) automaticamente
5. **Factura requiere RUC:** si el documento es de 11 digitos, es factura; si es de 8, es boleta
6. **Snapshot de precios:** se usa `precio_referencia` del producto al momento de la venta
7. **Precio de venta = precio_referencia:** no hay precio especial de venta, se toma el precio de referencia del producto
8. **Sin soft delete:** ventas se eliminan permanentemente

### Motor de permisos

**`User::tienePermiso(string $permiso): bool`**

Es el metodo **central de autorizacion**. Flujo:
1. Si el usuario no tiene `rol_id` asignado ‚Üí retorna `false`
2. Carga lazy la relacion `rol.permisos` si no estaba cargada
3. Verifica si la coleccion de permisos del rol contiene un permiso con `slug` igual al parametro
4. Retorna `true`/`false`

**Middleware `CheckPermission`** (`permiso`):
- Toma el slug del permiso desde la ruta
- Si `tienePermiso()` retorna `false` ‚Üí `abort(403, 'No tienes permiso...')`

**Middleware `EnsureUserIsActive`** (`activo`):
- Si el usuario autenticado tiene `activo = false`:
  - Cierra sesion, invalida session, regenera token CSRF
  - Redirecciona al login con error: *"Tu usuario esta inactivo. Contacta al administrador."*

**Flujo completo de autorizacion por ruta:**

```
Request ‚Üí [auth] ‚Üí [activo] ‚Üí [permiso:X] ‚Üí Controller
```

### Flujo: Crear usuario

1. Se validan campos:
   - `name`: **requerido**, string, max 255
   - `email`: **requerido**, string, email valido, max 255, **unico** en `users`
   - `password`: **requerido**, string, min 6, **confirmed** (requiere `password_confirmation`)
   - `rol_id`: **requerido**, integer, debe existir en `roles.id`
   - `activo`: opcional, boolean (default true)
2. Se crea el usuario con password hasheado
3. Redirecciona a index con mensaje de exito

---

### Flujo: Editar usuario

- Misma validacion que crear, excepto:
  - `password`: **opcional** (nullable), solo se actualiza si se proporciona un valor no vacio
  - `email`: validacion unica ignora el ID del usuario actual
- Se actualizan `name`, `email`, `rol_id`, `activo`

---

### Flujo: Activar/Desactivar usuario

- **Regla critica:** No se puede desactivar el propio usuario (`auth()->id() === $usuario->id`)
- Solo cambia el booleano `activo`
- Al desactivar, el usuario pierde acceso al sistema (el middleware `activo` lo redirige al login)

---

### Flujo: Gestion de roles

**Crear rol:**
- Valida `nombre` (requerido, unico), `descripcion` (opcional), `activo` (boolean), `permisos` (array de IDs)
- Crea el rol y sincroniza permisos via `sync()`

**Editar rol:**
- Misma validacion (ignora propio ID para unique name)
- Resincroniza permisos

**Eliminar rol:**
- **Regla critica:** No se puede eliminar un rol que tenga usuarios asignados (`$rol->usuarios()->exists()`)
- Si no tiene usuarios, se elimina
- Si se elimina un rol con `nullOnDelete`, los usuarios quedan con `rol_id = null` (sin permisos)

**Panel de roles inline (AJAX):**
- `GET /roles/panel-data` ‚Äî retorna JSON con todos los roles (con conteo de usuarios y permisos) y todos los permisos disponibles
- Las operaciones CRUD de roles responden tanto JSON como web

---

### Reglas de negocio clave

1. **Auto-desactivacion prohibida:** un usuario no puede desactivarse a si mismo
2. **Rol con usuarios no se elimina:**ÂøÖÈ°ªÂÖà desvincular usuarios
3. **Password opcional en edicion:** solo se actualiza si se proporciona
